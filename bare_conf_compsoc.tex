\begin{abstract}
Distributed Graph Representation Learning (GRL) on massive scale graphs suffers from the neighborhood explosion problem, requiring either costly synchronized network communication (e.g., DistDGL) or training in isolation on decoupled graph partitions. While decoupled training eliminates cross-worker runtime communication, it suffers from severe boundary accuracy degradation and sample scarcity in minor communities. This paper presents \textbf{EMO} (Executor-level Multi-community Orchestrator), a distributed systems framework that orchestrates decoupled GRL workloads natively on top of modern transactional Data Lakes (Delta Lake) using Apache Spark and YARN on AWS EMR. EMO models graph partitioning, boundary analysis, and community-aware auxiliary graph construction as standard relational database queries (joins, group-bys). By doing so, EMO can scale complex partitioned GNN workloads such as CAAN (Community-aware Auxiliary Networks, from past literature) with zero cross-worker training communication. EMO leverages Delta Lake's ACID transaction logs for experiment isolation, implements dynamic EBS storage redirection on EMR workers to prevent root-disk out-of-memory (OOM) crashes, and utilizes Apache Arrow for high-throughput JVM-Python data transfer. Our empirical evaluation on WikiCS and DeezerEurope datasets shows that EMO successfully scales community-decoupled GNNs, achieving up to a 2$\times$ accuracy boost over raw community GNNs by supporting CAAN auxiliary graphs, and competing with DistDGL in throughput on commodity cloud infrastructure.
\end{abstract}

\section{Introduction}
\subsection{Motivation \& Neighborhood Explosion in Distributed GRL}
Graph Neural Networks (GNNs) have emerged as the state-of-the-art paradigm for learning representation vectors from non-Euclidean graph data. However, scaling GNNs to web-scale social networks or transaction graphs is a fundamental systems challenge. The dependency of a node's representation on its recursive neighborhoods (neighborhood explosion) makes standard mini-batching extremely expensive in distributed clusters. State-of-the-art frameworks like DistDGL utilize active message-passing and parameter servers to synchronize weights and fetch node features across workers, resulting in massive network latency and high infrastructure costs.

\subsection{Decentralized Community-Based Training \& Accuracy Trade-Offs}
An alternative approach is decoupled, community-based GRL. By partitioning a graph into dense, localized communities (e.g., using Louvain or Label Propagation), a cluster can train independent GNNs on each partitioned community in parallel. Since the training takes place entirely within local worker nodes, this approach requires zero cross-worker network communication. 

However, community-decoupled GRL faces two major accuracy bottlenecks:
\begin{enumerate}
    \item \textbf{Boundary Accuracy Degradation}: Cutting inter-community edges isolates partition boundaries, causing nodes on the boundary of communities to lose topological context.
    \item \textbf{Sample Scarcity}: Graph community sizes naturally follow a power-law distribution. The presence of numerous tiny (``minor'') communities means that many local GNNs are trained on insufficient data, leading to severe overfitting.
\end{enumerate}

\subsection{The EMO Systems Architecture Solution}
To address these trade-offs, theoretical architectures like CAAN (Community-aware Auxiliary Networks) propose constructing auxiliary graphs: major communities are compressed into single ``super-nodes'' (represented by feature centroids) to maintain macro-level context, while minor communities are aggregated globally to maintain adequate training samples. Despite its theoretical promise, CAAN lacks a scalable, distributed systems architecture. Running such workflows currently requires custom ETL pipelines that manually export graphs from data warehouses to deep learning environments.

\subsection{Summary of Technical Contributions}
In this paper, we present \textbf{EMO} (Executor-level Multi-community Orchestrator), a distributed systems framework that bridges the gap between transactional data lakes and GRL training. EMO models the entire graph mining and GRL pipeline as relational queries on a Delta Lake data warehouse. The contributions of EMO are as follows:
\begin{itemize}
    \item \textbf{Data Lake Integration}: Unifies GRL storage and execution with transactional Delta Lake tables, enabling ACID isolation for ML runs and transaction log-based checkpoints.
    \item \textbf{Relational Graph Algebra}: Formulates graph partitioning, boundary tracking, and CAAN's auxiliary graph generation as standard Spark SQL relational joins and group-by aggregations.
    \item \textbf{Cloud-Native EMR Optimizations}: Implements dynamic local volume redirection on AWS EMR to prevent root EBS OOM crashes during CUDA/PyTorch execution, utilizes YARN for executor library sync, and leverages Apache Arrow for fast JVM-Python serialization.
    \item \textbf{Empirical Verification}: Demonstrates that EMO scales CAAN workloads on commodity CPU clusters, recovering boundary node accuracy by up to 2$\times$ over raw community GNNs.
\end{itemize}

\section{Related Work}
\subsection{Distributed GNN Training Systems}
Frameworks like DistDGL and PyG-Distributed scale GNNs via 1-hop or 2-hop neighborhood sampling across cluster nodes during mini-batching. While highly accurate, these systems rely on active, synchronized network calls to gather node features and aggregate gradients. This results in heavy communication overhead and requires complex, dedicated cluster setups. In contrast, EMO partitions the graph first, executing decoupled training with zero network communication.

\subsection{Graph Partitioning \& Community Detection}
Decoupled GNN training relies on graph partitioning to group nodes. METIS is the standard for producing balanced cuts but is computationally expensive to run on massive graphs. Scalable alternatives include Label Propagation (LPA), which runs natively in Spark, and Louvain community detection, which yields higher quality structural communities but has higher driver memory requirements. EMO supports modular plug-in algorithms for partitioning.

\subsection{Transactional Data Lakes in Machine Learning}
Modern data architectures use transactional formats like Delta Lake, Apache Hudi, or Apache Iceberg on top of cloud object storage (e.g., S3). These formats provide ACID transactions, time-travel history, and schema enforcement. EMO is the first framework to leverage Delta Lake storage contracts specifically to enforce experiment isolation and metadata checkpointing for distributed GRL.

\subsection{Delta Lake Transactional Schema \& ACID Checkpoints}
By using Delta Lake as the transactional backend, EMO establishes an isolation contract:
\begin{equation}
\text{Path}_{P2} = \text{delta-data/}\{dataset\}/\text{phase2\_nodes}/\{\text{exp\_id}\}_{dataset}_{alg}
\end{equation}
Different experiment runs (with different parameters or algorithms) execute in parallel without write collisions. Furthermore, checkpointing is achieved by querying the Delta log (\verb|_delta_log|) using a fast limit-1 SQL query. If the target directory has a valid transaction log, EMO skips that phase, avoiding expensive graph re-computations.

\subsection{Multi-Community Partitioning \& Subgraph Isolation}
During Phase 2, EMO segregates internal nodes from boundary nodes. Internal nodes interact exclusively within their assigned community, while boundary nodes are tracked for auxiliary network augmentation.

\section{Relational Graph Operations \& CAAN Workload}
A key system contribution of EMO is representing graph boundary analysis and CAAN's auxiliary topology as relational database operations.

\subsection{Expressing Boundary Operations in PySpark Relational Algebra}
To identify boundary nodes (nodes having at least one neighbor in a different community), EMO performs a relational join between the edge table and the node-community mapping, comparing source and destination community IDs:
\begin{equation}
\text{is\_boundary}(v) \iff \text{Deg}_{\text{orig}}(v) > \text{Deg}_{\text{intra}}(v)
\end{equation}
This is computed efficiently in Spark using SQL group-bys over the partitioned edge tables.

\subsection{Super-Node Abstraction \& Global Graph Topology}
For each major community $C_i$ (size $\ge$ $K$), EMO compresses other major communities $C_j$ ($j \neq i$) into super-nodes. The feature vector of the super-node is defined as:
\begin{equation}
f_{\text{super}}(C_j) = \frac{1}{|C_j|} \sum_{v \in C_j} f_v
\end{equation}
EMO executes this centroid calculation via a Spark SQL \verb|groupBy("community_id").mean("features")| aggregation. These super-node features, along with individual nodes from minor communities ($C < K$), are broadcasted to the executors.

\subsection{Vectorized Arrow-PyTorch Execution via PySpark UDFs}
Decoupled training is orchestrated using Spark's `applyInPandas` UDF. In each executor worker, PySpark receives the local nodes, super-nodes, and minor-node topology. Apache Arrow is used to serialize this tabular data directly into PyTorch/DGL tensors, bypassing Java-Python serialization bottlenecks.

\section{AWS EMR Cluster Optimizations}
Deploying deep learning on Apache Spark YARN clusters presents major systems challenges, which EMO resolves through three key optimizations:

\subsection{JVM Heap vs. PyTorch Off-Heap Memory Allocation}
PyTorch and DGL allocate GPU or CPU tensors off-heap via C++ memory managers. Standard JVM heap settings do not account for this, leading to YARN container terminations. EMO configures high memory overhead ratios:
\begin{lstlisting}[basicstyle=\footnotesize\ttfamily,columns=fullflexible]
spark.executor.memory = 28g
spark.executor.memoryOverhead = 12g
\end{lstlisting}

\subsection{Preventing EBS Root Disk OOM via Dynamic Volume Scanning}
EMR nodes typically have small (10 GB) root volumes (`/`). During GNN execution, PyTorch compiles CUDA kernels or downloads libraries, defaulting cache and temp paths to `/tmp` and filling up the root drive. EMO implements a dynamic scanning utility that finds attached storage partitions with the largest free capacity (e.g., `/mnt/tmp`, `/mnt1/tmp`) and overrides the environment variables:
\begin{lstlisting}[basicstyle=\footnotesize\ttfamily,columns=fullflexible]
os.environ['HOME'] = large_tmp
os.environ['TMPDIR'] = large_tmp
\end{lstlisting}

\subsection{YARN Executor Package Synchronization}
To ensure PyTorch and DGL are present in YARN containers without pre-imaging cluster nodes, EMO utilizes PySpark YARN bootstrap sync APIs (`sc.install_pypi_package`) combined with dynamic user-site search path patching (`sys.path.insert`).

\section{Discussion \& Future Database Extensions}
\subsection{Incremental Time-Travel GRL}
Enterprise graphs constantly change. Since Delta Lake logs every write operation via transaction logs, we can utilize Time Travel queries (`AS OF`) to identify which communities have had significant structural or feature changes, triggering retraining *only* for the affected executors.

\subsection{Multi-Dimensional Z-Ordering for S3 Graph Layouts}
We plan to Z-Order the nodes and edges tables in Delta Lake by \verb|community_id| and \verb|is_boundary|. This physical layout clustering on S3 ensures that executor workers load their target partitions with minimal S3 network requests, improving throughput.

\section{Conclusion}
We presented EMO, a distributed systems framework that orchestrates Graph Representation Learning workloads over transactional Data Lakes (Delta Lake). By mapping community partitioning, boundary tracing, and community-aware auxiliary networks into relational queries on Spark, EMO enables scalable, communication-free GNN training on commodity cloud clusters.
